Introduction

What is FHIR?

It's an acronym for Fast HealthCare Interoperability Resources It is an open specification aimed at data transmission, modeling, and serialization, designed for rapid understanding and implementation

FHIR Versions

What does FHIR define?

With this specification, we define resources similarly to how URLs are defined on the internet.

These URLs allow us to organize resources in a tree structure. Alt+Click for zoom.

FHIR Resource List

Resources

FHIR has extensive documentation, mainly focused on defining resources

Through it, we can learn the most important aspects of a resource. The content of the resource, examples of it (in XML, JSON, or Turtle), detailed descriptions of elements, mappings to other technologies, etc.

Resources

If we explore the documentation further, we can see a list of resource elements, annotations, cardinality (if the number of elements is zero, one, etc.), the data type of each element, and a description that includes constraints.

Data Types

These are the building blocks of the specification

  1. Primitive Data

    2. These are like data types in any programming language; they serve as the foundation for other data types.

    They are written in minorCamelCase, which means the first letter is lowercase and for the rest of the words, the first letter is uppercase.

Example: decimal

Data Types

These are the building blocks of the specification

  1. Primitive Data
  2. General Purpose Data

    3. These are types created from primitive types and/or others of the same kind

    They are written in MajorCamelCase, meaning the first letter of each word is uppercase.

Example: Quantity

Data Types

These are the building blocks of the specification

  1. Primitive Data
  2. General Purpose Data
  3. Metadata Data

    4. These are data types often used with metadata resources

    They are written in MajorCamelCase.

Example: ContactDetail

Extensions

Another thing that FHIR defines is how to extend the specification.

It sets rules for extending the specification

It allows anyone to create new resources and customize them, adding new functionalities to the core of FHIR

Implementation Guide Registry

The Core and Implementation Guides

Implementation Guides (IG) are the way an organization specifies its data model using the FHIR core.

An example of this is the SurPass implementation guide

What to Do with FHIR?

FHIR not only defines the model but also the way data transactions are conducted

Advantages of FHIR

It uses a REST architecture with minimal modifications, making it easy to learn and implement It extends the REST vocabulary, allowing more sophisticated resource searches It is a communication and modeling standard that gives us everything we need to build our application Provides toolkits for different programming languages, which helps a lot in implementing FHIR

Documentation of the RESTful API

HTTP Actions

FHIR uses the HTTP protocol for data transmission

HTTP has a set of methods to handle data transmission GET, POST, PUT, and DELETE are the most important

HTTP Actions

Types of verbs or actions

GET Used to retrieve a resource Equivalent to Read in the CRUD paradigm (Create-Read-Update-Delete) GET http://myfhirserver/url/of/resource

HTTP Actions

Types of verbs or actions

POST Used to create a new resource Equivalent to Create in the CRUD paradigm (Create-Read-Update-Delete) POST http://myfhirserver/url/of/resource

HTTP Actions

Types of verbs or actions

PUT Used to update a resource Equivalent to Update in the CRUD paradigm (Create-Read-Update-Delete) PUT http://myfhirserver/url/of/resource

HTTP Actions

Types of verbs or actions

DELETE Used to delete a resource Equivalent to Delete in the CRUD paradigm (Create-Read-Update-Delete) DELETE http://myfhirserver/url/of/resource

FHIR Actions

Remember that FHIR resources are organized like URLs

FHIR Actions

With FHIR, we can perform actions at different levels

Actions can affect the base (entire system), the level (all resources of a certain type), or instance (a particular resource).

FHIR Actions

To do this, we use the URL

VERB [base]/[level]/[instance] GET http://myfhirserver/baseR4/patient/63u62748493t237

FHIR Actions

The REST API of FHIR allows us to perform these actions:

FHIR Actions

Acting at the instance level

Name Description Schema Example
read Reads the current state of a resource GET [base]/[type]/[id] {?_format=[mime-type]} GET http://hapi.fhir.org/baseR4/Patient/2653234
vread Reads the state of a resource at a specific point in time GET [base]/[type]/[id]/_history/[vid] {?_format=[mime-type]} GET http://hapi.fhir.org/baseR4/Patient/2653234/_history/1
update Changes an existing resource (or creates one if it does not exist) PUT [base]/[type]/[id] {?_format=[mime-type]} PUT http://hapi.fhir.org/baseR4/Patient/2653234 [BODY --> Complete PatientObject]
patch Changes an existing resource by providing only the updates PATCH [base]/[type]/[id] {?_format=[mime-type]} PATCH http://hapi.fhir.org/baseR4/Patient/2653234 [BODY --> Parts of PatientObject]
delete Deletes a resource DELETE [base]/[type]/[id] DELETE http://hapi.fhir.org/baseR4/Patient/2653234
history Gets the history of a resource GET [base]/[type]/[id]/_history{?[parameters]&_format=[mime-type]} GET http://hapi.fhir.org/baseR4/Patient/2653234/_history
